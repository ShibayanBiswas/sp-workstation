# Authentication and Account Recovery

## Summary

Authentication uses two factors:

1. Registered email and password (roster emails in `src/data/team.ts` only).
2. A six-digit one-time password **generated by the system and shown on screen**.

No email delivery is used. Passwords are stored only as bcrypt hashes. Browser
state is held in signed, HTTP-only cookies.

## Login error handling

| Situation | API message | UI |
|-----------|-------------|-----|
| Email not in team roster | `Invalid email ID.` | Disclaimer popup |
| Valid email, wrong password | `Wrong password.` | Disclaimer popup |
| Malformed email | `Invalid email ID.` | Inline error |

Only approved `@rathi.com` addresses listed in `src/data/team.ts` can sign in.

## Sign-in sequence

```mermaid
sequenceDiagram
    participant User
    participant Login as POST /api/auth/login
    participant DB as MongoDB
    participant OtpPage as /otp
    participant Verify as POST /api/auth/verify-otp

    User->>Login: Email and password
    Login->>DB: Find User and compare bcrypt hash
    Login->>DB: Store 10-minute OTP
    Login-->>User: sp_pending cookie + otp in response
    User->>OtpPage: Code displayed on screen
    User->>Verify: Enter code
    Verify->>DB: Validate and consume OTP
    Verify-->>User: sp_session; clear sp_pending
```

### Cookies

- `sp_pending`: signed JWT, HTTP-only, 10 minutes (`login` or `password_reset`).
- `sp_session`: signed JWT, HTTP-only, 12 hours.

### Failed OTP behavior

Incorrect OTP returns `401`. The user retries on `/otp` while `sp_pending` is
valid. Expired pending sessions redirect to `/login`.

## Password change sequence

### From login (unauthenticated)

1. User clicks **Change password** on the login page.
2. `POST /api/auth/forgot-password` with email → OTP stored, `sp_pending` with
   `password_reset` purpose, code returned in API response.
3. User lands on `/change-password`, enters code + new password.
4. `POST /api/auth/change-password` validates OTP, updates hash, clears cookies.
5. User signs in with the new password.

Non-roster emails receive `Invalid email ID.` with a disclaimer popup.

### From dashboard (authenticated)

Logged-in users can open `/change-password` directly. The page calls
`POST /api/auth/request-password-otp` to generate a code for the active session.

Passwords must have at least eight characters with uppercase, lowercase, and a
number.

## User provisioning

Roster: `src/data/team.ts`. Passwords from `SEED_DEFAULT_PASSWORD_MAP` or
`scripts/seed-passwords.local.json` (gitignored).

## Security checklist

- Use a randomly generated `JWT_SECRET` (32+ bytes).
- Never commit `.env.local` or seed password files.
- Rotate default passwords after first sign-in.
- Restrict the seed endpoint after provisioning.
- OTP codes are visible on screen — deploy only on trusted internal networks.
